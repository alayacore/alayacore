# Bug Fix: Session Loading Only Renders Last Window

**Date**: 2026-03-20

## Issue

When loading a session from file (e.g., `~/alayacore-history-fix.md`), only the last window was rendered. All previous windows were missing from the display, even though the session file was correctly parsed and all messages were loaded.

## Root Cause

The bug was in the `markDirty` function in `internal/adaptors/terminal/window.go`. The incremental rendering optimization introduced in commit 1b9f63162 had a subtle logic error.

### The Original Buggy Code

```go
func (wb *WindowBuffer) markDirty(idx int) {
    if wb.dirtyIndex >= 0 && wb.dirtyIndex != idx {
        wb.dirtyIndex = fullRebuild
    } else {
        wb.dirtyIndex = idx  // BUG: Overwrites fullRebuild!
    }
}
```

### What Happened During Session Loading

When a session is loaded, `displayMessages()` rapidly creates multiple windows (one per message/tool). The `dirtyIndex` state evolved like this:

```
Window 0 created: dirtyIndex = 0
Window 1 created: dirtyIndex = fullRebuild (-2)  // Different window dirty
Window 2 created: dirtyIndex = 2                   // BUG: Overwrites fullRebuild!
Window 3 created: dirtyIndex = fullRebuild (-2)
Window 4 created: dirtyIndex = 4                   // BUG: Overwrites fullRebuild!
```

The `else` branch unconditionally set `dirtyIndex = idx`, even when it was already set to `fullRebuild`. This meant:

1. Windows 0-3 were marked for full rebuild
2. But then the flag was overwritten to only render the last window
3. When `ensureLineHeights()` ran, `dirtyIndex >= 0` triggered incremental update for just window 4
4. Windows 0-3 were never rendered!

### Why This Was Hard to Find

1. **Session file was correct** - The TLV encoding and parsing worked perfectly
2. **Message parsing was correct** - All messages were correctly loaded into memory
3. **Only display was broken** - The rendering cache was incorrectly updated
4. **Intermittent in appearance** - Only affected session loading (rapid window creation), not live streaming (same window repeatedly)

## The Fix

```go
func (wb *WindowBuffer) markDirty(idx int) {
    if wb.dirtyIndex == fullRebuild {
        // Already marked for full rebuild, keep it
        return
    }
    if wb.dirtyIndex >= 0 && wb.dirtyIndex != idx {
        // Different window already dirty - need full rebuild
        wb.dirtyIndex = fullRebuild
    } else {
        // Either clean (-1) or same window - mark just this one
        dirtyIndex = idx
    }
}
```

The key insight: **preserve the `fullRebuild` flag once set**. The early return ensures that once we know we need a full rebuild, we don't accidentally downgrade to a single-window update.

## Behavior After Fix

### Session Loading (Multiple Windows Rapidly)
```
Window 0 created: dirtyIndex = 0
Window 1 created: dirtyIndex = fullRebuild (-2)  // Different window dirty
Window 2 created: dirtyIndex = fullRebuild (-2)  // Preserved!
Window 3 created: dirtyIndex = fullRebuild (-2)  // Preserved!
Window 4 created: dirtyIndex = fullRebuild (-2)  // Preserved!
```
Result: Full rebuild renders all windows correctly.

### Delta Streaming (Same Window Repeatedly)
```
Window 0 created: dirtyIndex = 0
Delta to window 0: dirtyIndex = 0  // Same window, incremental OK
Delta to window 0: dirtyIndex = 0  // Same window, incremental OK
Delta to window 0: dirtyIndex = 0  // Same window, incremental OK
```
Result: Incremental update for O(1) performance (optimization still works).

## The Lesson

When implementing state-machine-like logic with special sentinel values (like `fullRebuild = -2`), always check if the sentinel is already set before overwriting. The bug was assuming the `else` branch was "safe" when it could actually overwrite an important state.
